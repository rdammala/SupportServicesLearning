🔄 Async/Await: The Foundation

Modern .NET is async-first. Every I/O operation (database, HTTP, file) should be async to free threads for other requests.

Core Pattern

// Async method returns Task
public async Task GetRefundAsync(string id)
{
    // await releases the thread until operation completes
    var item = await _container.ReadItemAsync(id);
    return item.Resource;
}

❌ Common Mistakes

  • .Result / .Wait(): Blocks the thread, defeats the purpose
  • Fire-and-forget (not awaiting): Exceptions get swallowed
  • Async void: Only use for event handlers, never for methods you call

✅ Best Practice

// 1. Return Task or Task, never void
// 2. Always await
// 3. Use ConfigureAwait(false) in libraries
public async Task ProcessRefundAsync(string id)
{
    var refund = await GetRefundAsync(id).ConfigureAwait(false);
    await _serviceBus.SendAsync(refund).ConfigureAwait(false);
}

📋 Records & Nullable Reference Types

Records (C# 9+): Immutable by default

// Contract models are records (immutable, value semantics)
public record RefundRequest(
    string Id,
    string UserId,
    decimal Amount,
    DateTime CreatedAt
);

// Deconstruct for convenience
var (id, userId, amount, _) = refund;

Nullable Reference Types (NRT)

// Enable in .csproj: enable

// Non-nullable by default (null = error)
public string Name { get; set; }  // must be set; null = compiler error

// Nullable explicitly marked
public string? Description { get; set; }  // can be null

// Null-check with pattern matching
if (refund.Description is not null)
{
    logger.LogInformation("Description: {desc}", refund.Description);
}

Benefits

  • Compiler catches null dereferences at build time
  • Clearer contracts ("is this nullable?")
  • Fewer NullReferenceException surprises at runtime

🔍 LINQ: Language Integrated Queries

Query vs Method Syntax

var results = _refunds
    .Where(r => r.Amount > 100)
    .OrderByDescending(r => r.CreatedAt)
    .Select(r => new { r.Id, r.Amount })
    .ToList();

// Equivalent query syntax
var results2 = (from r in _refunds
                where r.Amount > 100
                orderby r.CreatedAt descending
                select new { r.Id, r.Amount })
                .ToList();

Lazy Evaluation (IEnumerable)

// IEnumerable: lazy (not evaluated until enumerated)
IEnumerable query = _refunds
    .Where(r => r.Status == "Approved");
    // Nothing evaluated yet!

// ToList(): forces evaluation immediately
var results = query.ToList();  // NOW evaluated

Common Patterns

Pattern Example
Grouping .GroupBy(r => r.UserId).Select(g => new { UserId = g.Key, Count = g.Count() })
Join .Join(users, r => r.UserId, u => u.Id, (r, u) => new { Refund = r, User = u })
Aggregate .Aggregate(0, (sum, r) => sum + r.Amount)
Distinct .Select(r => r.UserId).Distinct().ToList()

🔧 Dependency Injection Patterns

Constructor Injection (Standard)

public class RefundService
{
    private readonly ICosmosContainer _container;
    private readonly IServiceBus _serviceBus;
    
    public RefundService(ICosmosContainer container, IServiceBus serviceBus)
    {
        _container = container;
        _serviceBus = serviceBus;
    }
}

Service Registration (Composition Root)

// In Startup or Program.cs
services.AddScoped();
services.AddSingleton();
services.AddTransient();

Lifetimes

Lifetime Use when
Transient Stateless, lightweight (created per injection)
Scoped Per-request context (default for web requests)
Singleton Shared across app lifetime (thread-safe, immutable)

💬 Interview Q&A

Q1: What's the difference between Task.Run and async/await?

A: Task.Run schedules work on a threadpool thread (useful for CPU-bound work). Async/await, when awaiting I/O, releases the current thread immediately without blocking (useful for scalability). For I/O-bound operations, use async/await; for CPU work, use Task.Run.

Q2: Why is nullability important in modern .NET?

A: Nullable Reference Types (NRT) let the compiler catch null dereferences at build time instead of runtime. This shifts errors left, reduces NullReferenceExceptions in production, and makes API contracts clearer (is this param nullable or not?).

Q3: LINQ Where vs Select: which is faster?

A: Neither is "faster"—both are O(n). But Where filters early (fewer items to process downstream), so it's often more efficient in practice. The real win is query composition: lazy evaluation means we only do work we need.

Q4: Why use records instead of classes?

A: Records are immutable by default (fewer bugs), have value semantics (two records with same data are equal), and support deconstruction. Classes are mutable and have reference semantics. Use records for DTOs/contracts, classes for stateful services.

Q5: What's the gotcha with Singleton + DI?

A: Singletons live for the app's lifetime. If you inject a Scoped service into a Singleton, the Scoped service gets promoted to Singleton too (captive dependency). This can cause resource leaks. Always prefer Transient → Scoped → Singleton in DI graphs.
