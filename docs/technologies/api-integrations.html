API Integrations: Service-to-Service Communication

Typed HTTP clients, auth handlers, resilience policies, and contract versioning for reliable inter-service calls

🔗 Typed HTTP Clients

Never use raw HttpClient. Wrap API calls in a typed client that:

  • Encodes base URL, headers, authentication
  • Defines request/response contracts
  • Applies resilience policies (retries, timeouts)
  • Maps HTTP errors to domain exceptions

Pattern: Typed Client in DI

// Service interface
public interface IRefundApiClient
{
    Task GetRefundAsync(string id);
    Task CreateRefundAsync(RefundRequest request);
}

// Implementation
public class RefundApiClient : IRefundApiClient
{
    private readonly HttpClient _httpClient;
    
    public RefundApiClient(HttpClient httpClient)
    {
        _httpClient = httpClient;
    }
    
    public async Task GetRefundAsync(string id)
    {
        var response = await _httpClient.GetAsync($"/refunds/{id}");
        return await MapResponseAsync(response);
    }
}

// Register with resilience policy
services.AddHttpClient(client =>
{
    client.BaseAddress = new Uri("https://refunds-api.azurewebsites.net");
    client.DefaultRequestHeaders.Add("User-Agent", "SupportServices/1.0");
})
.AddPolicyHandler(GetRetryPolicy())
.AddPolicyHandler(GetTimeoutPolicy());

🔐 Authentication Handlers

Use DelegatingHandler to inject auth tokens into every request without repeating code:

// DelegatingHandler: intercepts every HTTP request
public class BearerTokenHandler : DelegatingHandler
{
    private readonly ITokenProvider _tokenProvider;
    
    public BearerTokenHandler(ITokenProvider tokenProvider)
    {
        _tokenProvider = tokenProvider;
    }
    
    protected override async Task SendAsync(
        HttpRequestMessage request, 
        CancellationToken cancellationToken)
    {
        var token = await _tokenProvider.GetTokenAsync();
        request.Headers.Authorization = new AuthenticationHeaderValue("Bearer", token);
        return await base.SendAsync(request, cancellationToken);
    }
}

// Register
services.AddHttpClient()
    .AddHttpMessageHandler();

Service-to-Service (S2S) Patterns

  • XToken (Xbox Live 3.0): Mutual TLS + signed request token
  • Managed Identity: Azure AD token via IMDS (no secrets to rotate)
  • API Key: Simple but less secure; rotate regularly

💪 Resilience Policies

Use Polly library to handle transient failures gracefully:

Retry Policy

// Retry 3 times with exponential backoff
var retryPolicy = Policy
    .Handle()
    .Or()
    .OrResult(r => (int)r.StatusCode >= 500)
    .WaitAndRetryAsync(
        retryCount: 3,
        sleepDurationProvider: attempt => 
            TimeSpan.FromSeconds(Math.Pow(2, attempt))
    );

Timeout Policy

var timeoutPolicy = Policy.TimeoutAsync(TimeSpan.FromSeconds(5));

Circuit Breaker (stop hammering a failing service)

var circuitBreakerPolicy = Policy
    .Handle()
    .OrResult(r => (int)r.StatusCode >= 500)
    .CircuitBreakerAsync(
        handledEventsAllowedBeforeBreaking: 5,
        durationOfBreak: TimeSpan.FromSeconds(30)
    );

🔄 Contract Versioning

APIs evolve. Plan for backwards compatibility:

URL versioning (explicit, clear for clients)

GET /v1/refunds/{id}
GET /v2/refunds/{id}  // New fields, deprecated some old ones

Content negotiation (cleaner)

GET /refunds/{id}
Accept: application/vnd.supportservices.v2+json

Deprecation headers

// Response header warns consumers before shutdown
Deprecation: true
Sunset: Sun, 01 Jan 2027 00:00:00 GMT

Contract changes safely

  • ✅ Add new optional fields
  • ✅ New endpoints alongside old ones
  • ❌ Remove fields without warning period
  • ❌ Rename fields without mapping

💬 Interview Q&A

Q1: How do you prevent thundering herd (all retries at once)?

A: Exponential backoff with jitter: don't retry immediately, add randomness so all clients don't retry at the same millisecond. This prevents synchronized retries that overload the server.

Q2: Your circuit breaker trips. How does traffic recover?

A: Circuit breaker has states: Closed (normal) → Open (failing, rejects fast) → Half-Open (test single request). If half-open succeeds, transition to Closed. If fails, stay Open. This prevents cascading failures.

Q3: Should you retry on 4xx errors?

A: No. 4xx is client error (bad request, auth fail, not found). Retrying won't help; fix the request. Only retry 5xx and transient network errors.

Q4: How do you handle API version mismatches in production?

A: Version contract with deprecation windows. Warn 6 months before shutdown. Support v1 and v2 simultaneously during transition. Log if clients still use old version.
