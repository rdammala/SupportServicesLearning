Traces: See the full request

// One user request path:
GET /chats/123?refund_id=456
  ├─ Chat.Frontend
  │  └─ Get request latency: 200ms
  ├─ → Chat.Backend (RPC)
  │  └─ Process latency: 150ms
  ├─ → Cosmos DB query (GetChatById)
  │  └─ Query latency: 80ms
  ├─ → Service Bus publish (ChatCreated event)
  │  └─ Publish latency: 20ms

All linked by operation_Id (correlation ID)

Metrics: Understand patterns

  • Counter: Total requests, errors, items processed
  • Gauge: Current value (queue length, active connections)
  • Histogram: Distribution of latency, memory, RU cost

Logs: Point-in-time events

OpenTelemetry Instrumentation

Basic activity (span) with tags:

var activity = new Activity("GetChat").Start();
activity.SetTag("tenant.id", tenantId);
activity.SetTag("chat.id", chatId);
activity.SetTag("db.system", "cosmosdb");

try {
  var chat = await cosmos.GetChatById(chatId);
  activity.SetTag("http.status_code", 200);
  return chat;
} catch (Exception ex) {
  activity.SetTag("exception.type", ex.GetType().Name);
  activity.SetTag("http.status_code", 500);
  throw;
}

Custom metrics (meters):

// Counter: track events
meterProvider.CreateCounter("chats.created")
  .Add(1, new("tenant.id", tenantId));

// Gauge: track current state
meterProvider.CreateGauge("cosmos.active_connections")
  .Record(activeConnCount);

// Histogram: track distribution
meterProvider.CreateHistogram("cosmos.query_duration_ms")
  .Record(durationMs, new("operation", "GetById"));

RED & USE Metrics

RED (Request-driven services)

  • Rate: Requests per second
  • Errors: Failed requests (4xx, 5xx)
  • Duration: Latency distribution (p50, p95, p99)

USE (Resource-driven services)

  • Utilization: % resources in use (CPU, memory, Cosmos RU/s)
  • Saturation: Queue depth, backlog (waiting for resource)
  • Errors: Hardware/resource errors (OOMKill, connection refused)

Pro tip: Monitor percentiles (p95, p99), not averages. An average can hide spikes. "99th percentile latency < 500ms" is a better SLI than "average latency < 200ms".

KQL Queries

Query 1: Availability % by domain

requests
| where timestamp > ago(1h)
| where cloud_RoleName startswith "Chat"
| summarize 
    total = count(),
    success = countif(success == true)
    by cloud_RoleName
| extend availability_pct = (success * 100.0) / total
| project cloud_RoleName, availability_pct

Query 2: Error rate trend

requests
| where timestamp > ago(24h)
| where cloud_RoleName == "Chat.Frontend"
| summarize
    total = count(),
    errors = countif(success == false)
    by bin(timestamp, 5m)
| extend error_rate_pct = (errors * 100.0) / total
| render timechart

Query 3: Single request trace

union requests, dependencies, exceptions
| where operation_Id == "abc-123-def-456"
| project timestamp, type, name, duration, resultCode, message
| order by timestamp asc

Query 4: Dependency latency p99

dependencies
| where timestamp > ago(1d)
| where target == "cosmos.azure.com"
| summarize p99_duration = percentile(duration, 99)
    by name, target
| order by p99_duration desc

Dashboards & Alerts

SLO Dashboard structure

  • Top line: Current error budget remaining (% of budget, time remaining in month)
  • Charts: Availability %, latency p99, error rate over 30d
  • Burn rate: Fast window (1h) and slow window (6h) — if both alert, you have a real issue
  • Dependency health: Status of Cosmos, Service Bus, Key Vault
  • Capacity: Cosmos RU/s usage, App Service CPU/memory

Alert rule structure

Alert Name: "Chat.Frontend Error Rate > 1%"
Scope: Chat.Frontend
Condition: error_rate_pct > 1
Window: 5 minutes
Frequency: 1 minute
Severity: Sev-2
Action: PagerDuty webhook + Teams notification
Remediation: "See runbook: https://..."

💬 Interview Q&A

Q1: What's the difference between observability and monitoring?

A: Monitoring = looking at known metrics (uptime, CPU). Observability = ability to ask new questions about the system without modifying code. With observability, you ask "why did latency spike?" and find the answer from existing traces. With just monitoring, you're blind.

Q2: How would you correlate a user's failed request with a backend error?

A: Every request gets a correlation ID (operation_Id) at the entry point. It's passed downstream in headers: X-Correlation-ID or W3C traceparent. A union KQL query on operation_Id across requests, dependencies, exceptions shows the full call chain.

Q3: Your p99 latency spiked. How do you investigate?

A: I'd look at the latency distribution histogram (p50, p95, p99, max). If p99 spiked but p50 is fine, it's outliers. I'd query dependencies: which one is slow? Is it Cosmos (429 rate-limit?), network, or slow client? I'd look at traces for slow requests and find the slowest operation.

Q4: How do you set a good error budget alert?

A: Use multi-window burn-rate alerts: fast (1h window, alert if 14.4x burn rate) and slow (6h window, alert if 6x burn rate). Alert only when BOTH fire—that means you have a sustained, real issue, not a blip.

Q5: You emit 1 million metrics per minute. How do you not go broke?

A: Cardinality control: don't emit per-user or per-request metrics. Use dimensions (tenant, region, status) instead of creating new metric names. Aggregate at source before sending. Sample high-volume traces (keep 1% of successful requests, 100% of errors).
