Bicep & ARM Templates: Infrastructure as Code

Declarative Azure resource deployment with versioning, modular design, and GitOps workflows

📐 Bicep Basics

Bicep = modern syntax for Azure ARM templates. Compiles to ARM JSON but is far more readable.

Simple resource definition

// Storage account
resource storageAccount 'Microsoft.Storage/storageAccounts@2021-04-01' = {
  name: 'mystorageacct'
  location: 'eastus'
  kind: 'StorageV2'
  sku: {
    name: 'Standard_LRS'
  }
  properties: {
    accessTier: 'Hot'
  }
}

Outputs for downstream use

output storageAccountId string = storageAccount.id
output storageAccountName string = storageAccount.name

🧩 Modular Design

Break templates into reusable modules: networking, compute, storage, etc.

Module definition (network.bicep)

param location string = 'eastus'
param vnetName string

resource vnet 'Microsoft.Network/virtualNetworks@2021-02-01' = {
  name: vnetName
  location: location
  properties: {
    addressSpace: {
      addressPrefixes: ['10.0.0.0/16']
    }
    subnets: [...]
  }
}

output vnetId string = vnet.id

Use module (main.bicep)

module networking 'modules/network.bicep' = {
  name: 'networkingDeployment'
  params: {
    location: 'eastus'
    vnetName: 'prod-vnet'
  }
}

// Use output from module
output deployedVnetId string = networking.outputs.vnetId

⚙️ Parameters & Defaults

// Parameter with default
param environment string = 'dev'
param location string = resourceGroup().location

// Required parameter
param adminUsername string

// Allowed values
@allowed(['dev', 'staging', 'prod'])
param tier string

// Validation
@minLength(3)
@maxLength(24)
param storageAccountName string

✅ Best Practices

  • Use variables for computed values: Don't repeat expressions
  • Parameterize everything: Names, locations, counts, SKUs
  • Tag resources: For cost tracking and organization
  • Version modules: Treat bicep files like code
  • Use deployment stacks: Group related resources for cleanup
  • Test before deploy: `bicep validate` + what-if analysis

💬 Interview Q&A

Q1: ARM template vs Bicep—which should I use?

A: Always Bicep. It's cleaner syntax (no {} nesting hell), compiles to ARM, and is Microsoft's recommended path. ARM JSON is the compilation target, not the language you write.

Q2: How do you avoid deploying the same resource twice?

A: Use idempotent property names (storage account name includes timestamp or env). Bicep's `name` should be deterministic. Avoid name collisions by using naming conventions.

Q3: What's a deployment stack?

A: A group of related resources deployed together as a unit. When you delete the stack, all resources in it are deleted too. Prevents orphaned resources.

Q4: How do you version infrastructure changes?

A: Treat bicep files like code: version in git, review in PRs, tag releases. Use CI/CD to validate before deploy (what-if analysis).
