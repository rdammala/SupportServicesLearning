🏗️ The Reliability Architecture

The platform is multi-domain. Each domain has a Frontend (App Service, ASP.NET Core) and a Backend (Azure Functions, isolated worker), backed by Cosmos DB, Blob/Queue/Table storage, Service Bus, Key Vault, and Application Insights.

SRE reliability levers per layer

  • Front Door — health probes, WAF, failover routing between regions
  • App Service — staging/production slots, autoscale rules, health-check path /health
  • Functions — concurrency limits, retry policies, dead-letter handling
  • Cosmos — autoscale RU/s, partition-key hot-spot avoidance, PITR backup
  • Service Bus — lock renewal, max-delivery-count → dead-letter, prefetch tuning
  • Key Vault — secret/cert rotation, expiry alerts

SLIs, SLOs & Error Budgets

Definitions

  • SLI (Indicator): a measured number, e.g. proportion of requests < 500 ms
  • SLO (Objective): the target, e.g. 99.9% of requests succeed over 30 days
  • Error budget: 1 - SLO. At 99.9% you get 43m 12s of downtime/month
SLO Monthly error budget Weekly
99% 7h 18m 1h 41m
99.9% 43m 12s 10m 5s
99.95% 21m 36s 5m 2s
99.99% 4m 19s 1m

Defining an availability SLI in KQL

// Availability SLI: successful requests / total over 5m windows
requests
| where timestamp > ago(30d)
| where cloud_RoleName == "Chat.Frontend"
| summarize
    total = count(),
    good  = countif(success == true and duration < 1000)
    by bin(timestamp, 5m)
| extend availability = (good * 100.0) / total
| project timestamp, availability

Incident Response

The SRE on-call loop

When an alert fires, the SRE on-call follows this playbook:

🧪 Lab: Run a 30-minute incident drill
Scenario: Chat.Frontend latency spikes to 5s for 15 minutes
Task: Detect, triage, mitigate, postmortem
Acceptance: You can name 3 possible causes and 1 mitigation action per cause

Toil Reduction

Toil = operational work that is manual, repetitive, and doesn't add long-term value. SRE goal: keep toil < 50% of time.

Common toil in this platform

  • ⏱️ Rotating certificates (automated with Key Vault expiry alerts + Function trigger)
  • ⏱️ Slot swaps after deployments (automated in pipeline)
  • ⏱️ Scaling Cosmos during traffic spikes (automated with autoscale RU/s)
  • ⏱️ Manual rollbacks (automated with instant swap)
  • ⏱️ Creating backups (automated with PITR + weekly full snapshots)

💬 Interview Q&A

Q1: What's the difference between availability and reliability?

A: Availability is how much time the service is running (uptime %), reliability is whether it works correctly when it is running (error rate). A service can be 99% available but 50% reliable if half the requests fail.

Q2: How would you define SLOs for a multi-domain system?

A: Each domain has its own Frontend SLO based on customer impact. e.g., Chat.Frontend: 99.9% availability + p99 latency < 500ms. You aggregate across domains for a platform SLO. Asymmetric SLOs are OK — high-traffic domains can have tighter SLOs.

Q3: Your error budget is exhausted mid-month. What happens?

A: Feature freeze. No risky changes ship. Focus shifts to fixing reliability issues. This incentivizes developers to ship high-quality code and prevents a never-ending cycle of outages.

Q4: Walk me through a slot-swap rollback.

A: Production slot receives traffic. Staging has the new code. We validate staging (tests green, health-check path responds). We swap: now staging gets traffic, production is idle. If production (now idle) detects issues, we swap back instantly. Old code is still in memory on production, so rollback is < 1 second.

Q5: How do you correlate a failed user request with backend failures?

A: Each request gets a correlation ID (operation_Id) assigned at the gateway. We pass it through all downstream calls: Frontend → Backend → Cosmos → Service Bus. A union KQL query on operation_Id spans all logs/traces/metrics for that single request.
