The DevOps Engineer Perspective

From feature code to production: CI/CD, gates, safety, and shipping at scale

🧠 What You Own

DevOps engineers own the pipeline — the automated path from developer commit to production deployment:

  • ✅ Build pipeline (compile → test → lint → sign)
  • ✅ Infrastructure as Code (Bicep templates, deployment validation)
  • ✅ Release gates (approvals, test automation, health checks)
  • ✅ Deployment orchestration (blue-green slot swaps, canary, instant rollback)
  • ✅ Secrets management (Key Vault integration, rotation automation)
  • ✅ Artifact versioning (immutable artifacts, traceability)

Key practices:

  • Fast feedback: Parallel jobs (build + lint + format check)
  • No manual steps: Everything automated, repeatable
  • Immutable artifacts: Once built + signed, never rebuilt
  • Traceability: Build ID → commit SHA → deployed version

🛡️ Safety Mechanisms

  • Blue-green slots: Production and staging slots on App Service. Swap is atomic, rollback is instant.
  • Health gates: Staging must pass /health before approval prompt shows.
  • Approval gates: Manual step before production deployment. Requires reviewer sign-off.
  • Artifact signing: Ensures deployed code matches built code.
  • Bicep what-if: Preview changes before applying. Catch destructive changes.

💬 Interview Q&A

Q1: Walk me through a pipeline build failure. How do you unblock it?

A: I check the build logs. If it's a compile error, the dev fixes it and pushes again. If it's a test flake, we investigate: is it truly flaky or did the code break it? If lint fails (format/style), we run `dotnet format` locally and commit. The pipeline is a CI gate—it shouldn't block on infrastructure issues.

Q2: How do you handle secrets in the pipeline?

A: Secrets never appear in build logs. They live in Key Vault. The pipeline references them via variable groups (Azure Pipelines) or GitHub secrets (GitHub Actions). These are injected at runtime and masked in logs (shown as ***).

Q3: Your production slot swap failed. What do you do?

A: Swap back instantly by swapping slots again (production ← staging). This puts the old code back in production. Then investigate what went wrong: was the staging health check wrong? Did a dependency go down? Fix it, run smoke tests again, and re-attempt the swap.

Q4: How do you version artifacts?

A: Artifacts are versioned by build ID (incrementing counter) + git commit SHA. A deployment is traceable: you know exactly which code was built when, and which version is running in which slot.

Q5: What's the difference between "deploy" and "release"?

A: Deploy = put code on production infrastructure. Release = make it available to users. You can deploy 10 versions but keep showing version 9 to users (feature flags / traffic routing). This separates deployment risk from business risk.

✅ DevOps Readiness Checklist

Knowledge

  • ☐ I can name the 5 stages of a CI/CD pipeline

  • ☐ I can explain the difference between deployment and release
  • ☐ I know how to parameterize a Bicep deployment

  • ☐ I've triggered a pipeline manually and watched it complete
  • ☐ I've done a slot swap and verified traffic moved
  • ☐ I've rolled back a deployment in production
  • ☐ I've added a manual approval gate to a pipeline
  • ☐ I've read and interpreted build logs to debug a failure
