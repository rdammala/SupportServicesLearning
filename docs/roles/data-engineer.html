Cosmos DB Data Modeling

Core Principles

  • Model for access patterns, not normalization. Denormalize so a read = one query.
  • Partition key is the most important decision: choose high-cardinality, evenly-distributed, query-aligned keys. Avoid hot partitions (>50% RU on one key).
  • Contract vs storage models: API request/response models are separate from how data is stored. Map deliberately.
  • RU economics: Every read/write costs Request Units; batch and point-read where possible.

Partition Key Patterns

Pattern Use when
/tenantId Multi-tenant isolation, even tenant sizes
/userId Per-user data, many users
Synthetic /{type}_{date} Time-series with bounded hot range
Composite (hash of id) Avoiding monotonically increasing hotspots

Scenario: "Support conversation messages" container with queries by conversationId (recent-first)

Task:
1. List the top 3 access patterns
2. Choose a partition key and justify against hot-partition risk
3. Write the point-read and the range query

Acceptance: Partition key choice + 2 queries + one-line hot-partition justification

Example: Point read (cheapest)

// 1 RU: partition key + id
var msg = await container.ReadItemAsync(
    id: messageId, partitionKey: new PartitionKey(conversationId));

Example: Range query within partition

var query = new QueryDefinition(
    "SELECT * FROM c WHERE c.conversationId = @cid ORDER BY c.timestamp DESC")
    .WithParameter("@cid", conversationId);
    
var results = await container.GetItemQueryIterator(query)
    .ReadNextAsync();

💬 Interview Q&A

Q1: How do you choose a partition key?

A: I list the top access patterns (how I'll query the data), then pick a key that's high-cardinality (many distinct values), evenly distributed (no single tenant gets 50% of RUs), and query-aligned (I filter by it in my most common queries). If partition key X = tenantId, I can't efficiently query by userId without a cross-partition query (slower).

Q2: Your change-feed processor failed mid-batch. How do you prevent re-processing?

A: Idempotent tracking: when the processor handles a change, it records the Cosmos change-token + item ID in a processed-ids container. On retry, it checks: if this item was already processed, skip it. This ensures exactly-once semantics even with failures.

Q3: A KQL query is running slow on 1TB of data. How do you optimize?

A: Move the filter earlier: instead of SELECT * then filter, push filters into WHERE clauses at the top. Use materialized views if the query is repeated. Export hot data to ADX (Kusto) for faster analysis on large datasets. Sample data (percentiles) to avoid scanning 100% of rows.

Q4: Your telemetry volume is 1 billion events/day. How do you manage costs?

A: Cardinality control: use dimensions (tenant, region, status) instead of creating new metric names per user. Sample high-volume traces (keep 1% of successful requests, 100% of errors). Aggregate at source before sending to App Insights.

Q5: Explain contract vs. storage models.

A: Contract model = what API returns to clients (e.g., { userId, name, email }). Storage model = how data is stored in Cosmos (e.g., { userId, profile: { name, email, preferences } }). We map between them deliberately. This lets us change storage without breaking API clients.
