🧠 What Technical Support Owns

Support is the human bridge between a customer symptom and an engineering fix. You own accurate diagnosis, fast mitigation guidance, and clean escalation.

🔍 The Diagnosis Toolkit

Correlation IDs are everything

Every request gets an operation/correlation ID that flows App Service → Service Bus → Functions. Ask the customer (or capture from the response header) and pivot:

union requests, dependencies, exceptions, traces
| where operation_Id == ""
| project timestamp, itemType, name, success, resultCode, duration, message
| order by timestamp asc

This single query reconstructs the full journey of one failed request across services.

Scope: is it one customer or many?

exceptions
| where timestamp > ago(1h)
| where cloud_RoleName == "Refunds.Frontend"
| summarize affected = dcount(tostring(customDimensions.userId)),
            occurrences = count()
  by type, outerMessage
| order by occurrences desc

If affected is large → likely incident, escalate fast.
If = 1 → user-specific, dig into their data.

✅ Triage Runbook (First 10 minutes)

  1. Capture correlation ID, timestamp, customer/tenant, exact action
  2. Reproduce if possible; note the response code
  3. Run the union by operation_Id query
  4. Scope with the dcount affected query
  5. Check recent deploys (did a swap just happen?)
  6. Mitigate if known; else escalate with evidence package

📤 The Escalation Evidence Package

A good escalation = an engineer can act without asking you anything. Always include:

  • Correlation ID(s) — the smoking gun
  • Exact reproduction steps — "I click X, then Y, then error Z appears"
  • Scope — 1 customer vs many? Since when?
  • Telemetry artifacts — screenshot/export of the union query or exception
  • Response code/message — "502 Bad Gateway" vs "400 Bad Request" guides investigation
  • Timing — "Started 30 min ago after green deploy" → likely cause
  • Mitigation if attempted — "Cleared cache, retried, still fails"

❌ Bad escalation

"Customer says the app is broken. Can you look?"

✅ Good escalation

"Correlation ID XYZ: user johndoe@company gets 401 on POST /refunds at 2:15 PM UTC. Scope = 1 user. Reproduction: logged in, click Submit, error appears. Recent deploy: green Refunds.Frontend 2 PM. Exception: TokenExpiredException."

💬 Interview Q&A

Q1: A customer reports slowness, but telemetry shows normal latency. What do you do?

A: I ask: which page/operation? Check network tab (is it client-side rendering?) vs server latency. If server latency is normal but customer is slow, it's likely client-side JS, network conditions, or perception. I'd also check if it's region-specific (CDN issues) or browser-specific (cache problem).

Q2: Correlation ID shows a request went to Service, but Service says it never received it.

A: That's a Service Bus queue/topic lag or message loss scenario. I'd check: lease lag (is the processor catching up?), if the message was dead-lettered (moved due to repeated failure), or if there's a network split. Service Bus queuing discipline matters here.

Q3: You find a Cosmos 429 (throttled). How do you advise the customer?

A: First: is this widespread or isolated? If isolated, it's normal Cosmos backpressure—retry with exponential backoff. If widespread, Cosmos might be under-provisioned. Escalate to the SRE to increase RUs or review partition key distribution. Customer immediate action: none; retry is automatic.

Q4: How do you prevent "I thought it was fixed" loops?

A: Document the root cause and fix in our KB. Share with team in standup. Add a regression test if applicable. When escalating, reference prior incidents so we catch patterns early.

Q5: A customer says data went missing. How do you investigate?

A: Check: (1) Is the data actually gone or just not visible (soft delete, schema version mismatch)? (2) Change feed lag—did the processor fall behind? (3) If truly missing, pull the audit log (or change-feed history) to see what happened. Escalate with the evidence of whether it was a delete operation or a system failure.

✅ Support Engineer Readiness Checklist

Knowledge

  • ☐ I can write a correlation ID union query
  • ☐ I understand HTTP status codes (4xx = client, 5xx = server)
  • ☐ I can scope an issue (1 user vs many)
  • ☐ I know the symptom → cause playbook by heart
  • ☐ I understand what goes in an escalation email

Operational

  • ☐ I've traced a production request end-to-end
  • ☐ I've diagnosed a customer issue from correlation ID alone
  • ☐ I've escalated a real incident with evidence
  • ☐ I've written a KB article for a common issue
  • ☐ I've attended a postmortem and asked "what changed?"
