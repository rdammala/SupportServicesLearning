🧪 Labs & Capstone Scenarios

Hands-on, self-graded exercises and end-to-end scenario drills

How Labs Work

Lab Structure

Each lab:

  • 📝 Scenario — The problem you're solving
  • 🎯 Acceptance criteria — How you know you're done
  • ⏱️ Estimated time — 30 min to 4 hours
  • 💡 Hints — Nudges, not solutions
  • ✅ Self-grading checklist — Verify your work

Types of Labs

  • Inline labs — In each role/tech guide (3–5 per guide, 30–60 min each)
  • Capstone labs — End-to-end scenarios (2–4 hours, cross-role/cross-tech)
  • Incident drills — Timed operational exercises (30 min, fire-and-chaos scenario)

Capstone Labs (End-to-End Scenarios)

Each capstone is a 2–4 hour scenario spanning multiple roles and technologies. Self-graded against acceptance criteria.

Capstone 1: Ship, Observe, Recover

🚀 Full DevOps loop

Goal: Ship a .NET API to production, observe it with SLOs and dashboards, inject failures, and roll back safely.

Scenario

You're deployed a new Chat.Frontend feature to production. Your SRE teammate sets up an SLO dashboard. An alert fires: error rate spiked to 5%. You investigate via KQL, identify the cause, roll back, and postmortem.

Acceptance Criteria

  • ✓ Built a .NET API with 3 endpoints
  • ✓ Authored Bicep infra (App Service + Application Insights)
  • ✓ Created a CI pipeline (build → test → deploy → swap)
  • ✓ Instrumented code with OTel (spans + custom metrics)
  • ✓ Created SLO dashboard (latency + error rate + custom metric)
  • ✓ Set burn-rate alerts (slow + fast windows)
  • ✓ Ran a failure injection (killed a downstream service)
  • ✓ Rolled back in < 1 minute
  • ✓ Wrote a postmortem with 2 action items

Time: 3–4 hours

Capstone 2: Data + Observability

📊 Data pipeline + analytics

Goal: Model Cosmos data, build an idempotent change-feed processor, emit custom events, and query anomalies in KQL.

Scenario

The Refunds domain processes refund requests. You design the Cosmos partition key, write a Service Bus consumer with idempotent tracking, emit events to App Insights, and set up a KQL workbook for funnel analysis + anomaly detection.

Acceptance Criteria

  • ✓ Designed Cosmos DB schema (account, database, container, partition key)
  • ✓ Justified partition key choice (access pattern analysis)
  • ✓ Wrote idempotent Service Bus processor (duplicate detection by processed-id)
  • ✓ Emitted trackEvent with custom dimensions (tenant, status, amount)
  • ✓ Wrote 4 KQL queries (funnel, cohort, time-series, anomaly)
  • ✓ Created parameterized workbook
  • ✓ Set alert rule for anomalies

Time: 2–3 hours

Capstone 3: Agentic Support Assistant

🤖 AI + guardrails + evals

Goal: Expose a typed API client as an agent tool, add RAG grounding, implement safety guardrails, and build an evaluation harness.

Scenario

Build a support assistant agent that can look up customer orders and refunds (via mocked APIs), fetch relevant help articles (Azure AI Search hybrid search), answer questions with citations, and knows when to escalate to a human (guardrails).

Acceptance Criteria

  • ✓ Defined 2 tools (GetOrder, GetRefund) wrapping typed clients
  • ✓ Integrated Azure AI Search (hybrid search: keyword + vector)
  • ✓ Injected search results into prompt with citations
  • ✓ Implemented guardrails (max 5 iterations, confirmation for refunds, jailbreak detection)
  • ✓ Instrumented with OTel (model call spans, tool call spans, latency metrics)
  • ✓ Built eval harness (10 test cases, scorer function)
  • ✓ Ran continuous eval and tracked score over time

Time: 3–4 hours

Capstone 4: Container Platform Slice

☸️ Kubernetes + autoscaling + canary

Goal: Containerize a microservice, deploy to AKS with proper resource requests/limits, set up autoscaling, and do a canary rollout.

Scenario

The Search service needs to run on AKS for higher compute. You containerize it (multi-stage Dockerfile, non-root user), deploy to AKS with readiness/liveness probes, set up HPA (CPU-based) + KEDA (queue-based), and do a safe canary rollout with metrics validation.

Acceptance Criteria

  • ✓ Wrote multi-stage Dockerfile (build + runtime, < 500MB final image)
  • ✓ Deployed to AKS with resource requests (CPU/memory)
  • ✓ Set readiness probe (/health) + liveness probe (/alive)
  • ✓ Configured HPA (target CPU 70%, scale 2–10 replicas)
  • ✓ Configured KEDA (scale on Service Bus queue length)
  • ✓ Authored Bicep for Deployment + Service + Ingress
  • ✓ Ran load test and validated autoscaling
  • ✓ Performed canary rollout (10% → 50% → 100% traffic over 3 stages)
  • ✓ Verified metrics during canary (error rate stayed < 0.1%)

Time: 4 hours

Incident Drill (30 minutes, timed)

Scenario: Chat.Frontend outage

Time: 30 minutes

At 09:00 UTC: Alert fires: Chat.Frontend error rate 50%, latency p99 5s.

Your tasks:

  1. Declare severity (SEV-1, SEV-2, SEV-3) + page on-call team
  2. Triage (5 min): Use KQL union query by operation_Id to find the failing service
  3. Identify (10 min): Root cause (Cosmos hot partition, Service Bus deadlock, dependency timeout)
  4. Mitigate (5 min): Scale Cosmos RU/s, kill problematic consumer, or roll back
  5. Comms (5 min): Draft 3 status updates (incident start, investigating, resolved)
  6. Postmortem items (5 min): List 3 preventative actions (monitoring, testing, automation)

Success criteria:

  • ✓ Severity declared within 1 minute
  • ✓ Root cause identified within 12 minutes
  • ✓ Mitigation action taken by 18 minutes
  • ✓ Status updates are clear and concise
  • ✓ At least 1 postmortem action is concrete (automation, alert, test case)

How to Get Started

  1. Pick a role guide and read it (15 min)
  2. Do the inline labs in that guide (30–60 min each, 3–4 labs)
  3. Pick a capstone that resonates with you (2–4 hours)
  4. Do the incident drill for operational fluency (30 min)
  5. Build a portfolio — document what you learned and share on GitHub
