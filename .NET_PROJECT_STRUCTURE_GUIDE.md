## Project & Solution Files

### .slnx (Solution File - New Format)

**What It Is:**
Newer solution format introduced in .NET 9+ that replaces `.sln`. More concise, machine-generated, and designed for scalable monorepos.

**Why It Exists:**
- ✅ Cleaner syntax than `.sln` (XML-like, compact)
- ✅ Better for monorepos with many projects
- ✅ Easier to parse and generate programmatically
- ✅ Supports solution folders and filtering

**Who Creates It:**
- **Auto-generated:** `dotnet new sln --name MySolution`
- **Visual Studio:** When you create a new solution
- **OneBranch SDK:** Generates .slnx for SupportServices

**Example from SupportServices:**
```xml
<!-- SupportServices.slnx (Root) -->
<solution name="SupportServices" version="10.0">
  <folder name="Chat">
    <project path="Chat/Chat.slnx" />
  </folder>
  <folder name="Common">
    <project path="Common/Common.slnx" />
  </folder>
</solution>

<!-- Chat/Chat.slnx (Domain-level) -->
<solution name="Chat" version="10.0">
  <folder name="Frontend">
    <project path="Chat/Frontend/Frontend.csproj" />
    <project path="Chat/Frontend/Frontend.Tests/Frontend.Tests.csproj" />
  </folder>
  <folder name="Storage">
    <project path="Chat/Storage/Storage.csproj" />
  </folder>
</solution>
```

**Developer's Role:**
- **Usually:** Just reference it; don't edit manually
- **When to edit:** Add/remove project groups, organize solution folders
- **Pro tip:** Let Visual Studio maintain it; manual edits risk corruption

### .sln (Legacy Solution File)

**What It Is:**
Legacy solution format (pre-.NET 9). Plain text format that lists all projects in a solution.

**Why It Exists:**
- ✅ Legacy support for older .NET versions
- ✅ Compatible with older Visual Studio versions
- ✅ Simple text format for version control

**Auto-Generated vs Manual:**
- **Auto-generated:** `dotnet new sln`
- **Rarely edited manually:** Changes can break project references

**Example (Legacy Format):**
```
Microsoft Visual Studio Solution File, Format Version 12.00
# Visual Studio Version 17
VisualStudioVersion = 17.8.34124.123
MinimumVisualStudioVersion = 10.0.40219.1

Project("{9A19103F-16F7-4668-BE54-9A1E7A4F7557}") = "Chat.Frontend", "Chat\Frontend\Frontend.csproj", "{ABC123D0-EF12-3456-AB78-901234567890}"
EndProject

Project("{9A19103F-16F7-4668-BE54-9A1E7A4F7557}") = "Chat.Frontend.Tests", "Chat\Frontend\Frontend.Tests\Frontend.Tests.csproj", "{DEF456G0-HI78-9012-CD34-567890123456}"
EndProject

Global
	GlobalSection(SolutionConfigurationPlatforms) = preSolution
		Debug|Any CPU = Debug|Any CPU
		Release|Any CPU = Release|Any CPU
	EndGlobalSection
EndGlobal
```

**Status in SupportServices:**
- ✅ `.slnx` is the new standard
- ⚠️ Legacy `.sln` files still present for older projects
- 📌 Migration path: Use `.slnx` for new work

### .csproj (C# Project File)

**What It Is:**
Project file that defines how a single C# project builds — what files to include, dependencies, build settings, compiler options.

**Why It Exists:**
- ✅ Tells MSBuild what to compile and how
- ✅ Manages NuGet package references
- ✅ Defines output type (library, executable, test)
- ✅ Specifies target framework (.NET 10, .NET Framework, etc.)

**Who Creates It:**
- **Auto-generated:** `dotnet new classlib`, `dotnet new console`, etc.
- **IDE:** Visual Studio creates via templates
- **Developers:** Usually don't create manually; edit existing ones

**Modern .csproj (SupportServices Example):**
```xml
<!-- Chat/Frontend/Frontend.csproj -->
<Project Sdk="Microsoft.NET.Sdk.Web">

  <PropertyGroup>
    <!-- Target framework: .NET 10 -->
    <TargetFramework>net10.0</TargetFramework>
    
    <!-- Language version: C# 14 (latest) -->
    <LangVersion>latest</LangVersion>
    
    <!-- Enable nullable reference types (stricter null checking) -->
    <Nullable>enable</Nullable>
    
    <!-- Treat compiler warnings as errors (strict mode) -->
    <TreatWarningsAsErrors>true</TreatWarningsAsErrors>
    
    <!-- Generate assembly name from folder path (auto-derived) -->
    <!-- Namespace: Microsoft.Xbox.Support.Chat.Frontend -->
  </PropertyGroup>

  <!-- NuGet package references (versions from Directory.Packages.props) -->
  <ItemGroup>
    <PackageReference Include="Microsoft.AspNetCore.Mvc.Core" />
    <PackageReference Include="Microsoft.Extensions.DependencyInjection" />
  </ItemGroup>

  <!-- Project references -->
  <ItemGroup>
    <ProjectReference Include="../Storage/Storage.csproj" />
    <ProjectReference Include="../../Common/AspNetCore/AspNetCore.csproj" />
  </ItemGroup>

  <!-- Imports shared build logic -->
  <Import Project="../../Directory.Build.props" />
  <Import Project="../../Directory.Build.targets" />

</Project>
```

**Key Points:**
- ✅ **Minimal in modern .NET:** Uses Central Package Management (see Directory.Packages.props)
- ✅ **Auto-derived namespace:** From folder path via Directory.Build.props
- ✅ **No version attributes:** Versions in Directory.Packages.props instead
- ⚠️ **Never set `AssemblyName` or `RootNamespace`:** Auto-derived by build system

**What NOT to Put in .csproj:**
```xml
<!-- ❌ DON'T DO THIS -->
<PropertyGroup>
  <AssemblyName>Custom.Name</AssemblyName>  <!-- Wrong: auto-derived -->
  <RootNamespace>Custom.Namespace</RootNamespace>  <!-- Wrong: auto-derived -->
</PropertyGroup>

<ItemGroup>
  <PackageReference Include="Newtonsoft.Json" Version="13.0.0" />  <!-- Wrong: version in .props -->
</ItemGroup>
```

## Configuration Files

### .config Files (Application Configuration)

**What It Is:**
XML configuration files for .NET applications, particularly ASP.NET Core and Console apps.

**Types:**

**1. app.config (Desktop Apps)**
```xml
<!-- Classic .NET Framework app configuration -->
<configuration>
  <startup version="v4.0" />
  <connectionStrings>
    <add name="DefaultConnection" connectionString="Server=localhost;Database=MyDb" />
  </connectionStrings>
  <appSettings>
    <add key="LogLevel" value="Debug" />
  </appSettings>
</configuration>
```

**2. web.config (ASP.NET Core Legacy)**
```xml
<!-- Hosts ASP.NET Core app in IIS -->
<configuration>
  <system.webServer>
    <aspNetCore processPath=".\bin\Debug\net10.0\MyApp.exe" />
  </system.webServer>
</configuration>
```

**Status in SupportServices:**
- ⚠️ **Mostly obsolete:** Modern .NET uses `appsettings.json` instead
- ✅ **Still present:** For ASP.NET Core hosting in IIS
- 📌 **Trend:** Moving to `appsettings.json` + environment variables

### appsettings.json (Primary Configuration)

**What It Is:**
JSON file for application settings, database connections, logging levels, feature flags, etc.

**Why It Exists:**
- ✅ Human-readable (JSON vs XML)
- ✅ Typed configuration with Options Pattern
- ✅ Environment-specific variants (appsettings.Development.json)
- ✅ Can be overridden by environment variables

**Structure (SupportServices Example):**
```json
{
  "Logging": {
    "LogLevel": {
      "Default": "Information",
      "Microsoft": "Warning"
    }
  },
  "ConnectionStrings": {
    "CosmosDb": "AccountEndpoint=https://localhost:8081/;AccountKey=C2y6yDjf5/R+ob0N8A7Cgv30VRDJIWEHLM+4QDU5DE2nQ9nDuVTqwM=",
    "ServiceBus": "Endpoint=sb://localhost:5671/"
  },
  "Settings": {
    "OrderProcessingTimeout": 30,
    "MaxRetries": 3,
    "EnableFeatureX": true
  }
}
```

**Environment-Specific Variants:**
```
appsettings.json                  (Base settings)
appsettings.Development.json      (Dev overrides)
appsettings.Staging.json          (Staging overrides)
appsettings.Production.json       (Prod overrides — never committed)
```

**Who Creates It:**
- **Auto-generated:** When you create ASP.NET Core project
- **Developer maintains:** Base appsettings.json in repo
- **CI/CD replaces:** Prod settings via Key Vault/secrets

**Binding in Code (Options Pattern):**
```csharp
// appsettings.json
{
  "ChatSettings": {
    "MaxConcurrentUsers": 1000,
    "MessageQueueSize": 5000,
    "TimeoutSeconds": 30
  }
}

// Code
public class ChatSettings
{
    public int MaxConcurrentUsers { get; set; }
    public int MessageQueueSize { get; set; }
    public int TimeoutSeconds { get; set; }
}

// Startup
services.AddOptionsWithBinding<ChatSettings>();
// Now: IOptions<ChatSettings> chatSettings — injected everywhere

// Access in code
public class ChatService
{
    private readonly ChatSettings _settings;
    
    public ChatService(IOptions<ChatSettings> settings)
    {
        _settings = settings.Value;  // Fully typed, intellisense support
    }
}
```

## Build & Compilation Files

### Directory.Build.props (Centralized Build Properties)

**What It Is:**
Master build file that applies settings to ALL projects in a directory tree. Avoids duplication across hundreds of .csproj files.

**Location:** Repository root: `Directory.Build.props`

**Why It Exists:**
- ✅ **Single point of control:** Change once, applies everywhere
- ✅ **DRY principle:** Define common settings once
- ✅ **Consistency:** All projects use same compiler, language version, etc.
- ✅ **Scalability:** SupportServices has 385+ projects; impossible to manage individually

**SupportServices Example:**
```xml
<!-- c:\SupportServices\Directory.Build.props -->
<Project>

  <!-- Shared compiler settings -->
  <PropertyGroup>
    <LangVersion>latest</LangVersion>
    <Nullable>enable</Nullable>
    <TreatWarningsAsErrors>true</TreatWarningsAsErrors>
    <GenerateDocumentationFile>true</GenerateDocumentationFile>
  </PropertyGroup>

  <!-- Auto-derive namespace from folder path -->
  <PropertyGroup>
    <!-- Example: Refunds/Frontend/Frontend/ → Microsoft.Xbox.Support.Refunds.Frontend -->
    <RootNamespace>$(MSBuildProjectDirectory.Replace($([System.IO.Path]::GetFullPath("$(MSBuildThisFileDirectory)").Replace("\", "\\")), "").Replace("\", ".").TrimStart('.') |> %(Text.Insert(0, "Microsoft.Xbox.Support.")))</RootNamespace>
  </PropertyGroup>

  <!-- Code analyzers (on by default) -->
  <ItemGroup>
    <PackageReference Include="Microsoft.CodeAnalysis.NetAnalyzers" PrivateAssets="All" />
    <PackageReference Include="StyleCop.Analyzers" PrivateAssets="All" />
  </ItemGroup>

  <!-- Centralized dependency versioning -->
  <Import Project="$(MSBuildThisFileDirectory)Directory.Packages.props" />

</Project>
```

**Who Creates/Modifies:**
- **Created by:** Platform team (first time only)
- **Modified by:** Architects when changing global build rules
- **Developers:** Should rarely touch this
- **Sync:** Check before building after long absences

### Directory.Packages.props (Central Package Management)

**What It Is:**
Single file that declares ALL NuGet package versions across the entire monorepo. Individual .csproj files reference packages WITHOUT versions.

**Why It Exists:**
- ✅ **Version control:** One place to update Newtonsoft from 12.0 to 13.0
- ✅ **Prevents conflicts:** No "one project uses 13.0, another uses 12.0"
- ✅ **Security:** Update vulnerable packages globally
- ✅ **Scalability:** Crucial for 385+ projects

**SupportServices Example:**
```xml
<!-- Directory.Packages.props (Root) -->
<Project>
  
  <PropertyGroup>
    <!-- Enable central package versioning -->
    <ManagePackageVersionsCentrally>true</ManagePackageVersionsCentrally>
  </PropertyGroup>

  <ItemGroup>
    <!-- .NET Runtime packages -->
    <PackageVersion Include="Microsoft.AspNetCore.Mvc.Core" Version="10.0.0" />
    <PackageVersion Include="Microsoft.Extensions.Configuration" Version="10.0.0" />
    <PackageVersion Include="Microsoft.Extensions.DependencyInjection" Version="10.0.0" />
    
    <!-- Azure packages -->
    <PackageVersion Include="Azure.Cosmos" Version="4.0.0" />
    <PackageVersion Include="Azure.Storage.Blobs" Version="12.20.0" />
    <PackageVersion Include="Azure.Messaging.ServiceBus" Version="7.18.0" />
    
    <!-- Data/serialization -->
    <PackageVersion Include="Newtonsoft.Json" Version="13.0.3" />
    <PackageVersion Include="System.Text.Json" Version="10.0.0" />
    
    <!-- Testing -->
    <PackageVersion Include="Microsoft.NET.Test.Sdk" Version="17.9.0" />
    <PackageVersion Include="MSTest.TestFramework" Version="3.1.1" />
    <PackageVersion Include="Moq" Version="4.20.70" />
    
    <!-- Utilities -->
    <PackageVersion Include="FluentAssertions" Version="6.12.0" />
    <PackageVersion Include="Snapper" Version="3.0.0" />
  </ItemGroup>

</Project>
```

**How .csproj Files Reference Packages:**
```xml
<!-- Chat/Frontend/Frontend.csproj -->
<ItemGroup>
  <!-- ✅ NO VERSION SPECIFIED -->
  <!-- Version comes from Directory.Packages.props -->
  <PackageReference Include="Microsoft.AspNetCore.Mvc.Core" />
  <PackageReference Include="Azure.Cosmos" />
</ItemGroup>
```

**Who Creates/Modifies:**
- **Created by:** Platform team during repo setup
- **Modified by:** Dependency managers or architecture team
- **When:** Quarterly security updates, major version upgrades
- **Developers:** Never directly edit; request via PR

**Maintenance Process:**
```bash
# Check for outdated packages
dotnet outdated

# Update a package version
# Edit Directory.Packages.props, change version, then:
dotnet build --no-incremental

# Test all projects to ensure compatibility
dotnet test
```

### Directory.Build.targets (Post-Build Logic)

**What It Is:**
Companion to Directory.Build.props; runs AFTER all projects build. Useful for post-build steps, file copying, code generation.

**Example:**
```xml
<!-- Directory.Build.targets -->
<Project>

  <!-- Copy runtime files after build -->
  <Target Name="CopyRuntimeFiles" AfterTargets="Build">
    <Copy 
      SourceFiles="@(RuntimeFiles)" 
      DestinationFolder="$(OutputPath)" />
  </Target>

  <!-- Generate version info from git -->
  <Target Name="GenerateVersionInfo" BeforeTargets="GenerateAssemblyInfo">
    <Exec Command="git describe --always --dirty" ConsoleToMSBuild="true">
      <Output TaskParameter="ConsoleOutput" PropertyName="GitVersion" />
    </Exec>
    <ItemGroup>
      <AssemblyAttribute Include="System.Reflection.AssemblyInformationalVersion">
        <_Parameter1>$(GitVersion)</_Parameter1>
      </AssemblyAttribute>
    </ItemGroup>
  </Target>

</Project>
```

## Version Control Files

### .gitignore (Exclude Files from Git)

**What It Is:**
Text file listing patterns of files/folders Git should NOT track (temp files, build outputs, secrets, etc.).

**Why It Exists:**
- ✅ Don't commit `bin/`, `obj/` (build artifacts)
- ✅ Don't commit `.vs/` (Visual Studio cache)
- ✅ Never commit secrets (`appsettings.Production.json`, `.env`)
- ✅ Don't commit IDE-specific files (`.idea/`, `*.swp`)

**SupportServices Example:**
```
# Build outputs
bin/
obj/
*.dll
*.exe

# Visual Studio
.vs/
*.user
*.suo
*.sln.docstates

# IDE
.idea/
.vscode/
*.swp
*.code-workspace

# Runtime
*.log
*.tmp

# OS
.DS_Store
Thumbs.db

# Secrets (NEVER commit)
appsettings.Production.json
appsettings.Local.json
.env
secrets.json

# NuGet
packages/
.nuget/
*.nupkg

# Test results
TestResults/
coverage/

# Cosmos DB Emulator
CosmosDB/
emulator_data/
```

**Who Creates It:**
- **Auto-generated:** `dotnet new sln` creates default .gitignore
- **Platform team:** Customizes for monorepo needs
- **Developers:** Rarely modify; add patterns when discovering untracked files

**Pro Tips:**
```bash
# Check what's being ignored
git check-ignore -v <filename>

# See what Git would ignore
git status --ignored

# Accidentally committed something? Remove without deleting local file
git rm --cached <filename>
```

### .gitattributes (Git File Handling)

**What It Is:**
Controls how Git handles line endings, binary files, and merge strategies per file type.

**Why It Exists:**
- ✅ **Line endings:** Ensure CRLF (Windows) vs LF (Unix) consistency
- ✅ **Binary files:** Don't attempt to merge binary formats
- ✅ **Diff tools:** Use appropriate diff for file type (e.g., image files)

**SupportServices Example:**
```
# Auto detect line endings and normalize to LF
* text=auto

# C# files: always CRLF (Windows)
*.cs eol=crlf

# JSON/XML: CRLF
*.json eol=crlf
*.xml eol=crlf
*.yml eol=crlf

# Batch/PowerShell: CRLF
*.bat eol=crlf
*.ps1 eol=crlf

# Shell scripts: always LF (Unix)
*.sh eol=lf
*.bash eol=lf

# Binary files: don't merge
*.png binary
*.jpg binary
*.gif binary
*.dll binary
*.exe binary

# Documentation: text
*.md text eol=lf
```

**Who Creates It:**
- **Platform team:** Sets up during repo initialization
- **Developers:** Rarely modify

**Pro Tips:**
```bash
# Check what attributes apply to a file
git ls-files --stage <filename>

# Refresh line endings after adding .gitattributes
git add -u
git commit -m "Normalize line endings"
```

## CI/CD & Infrastructure Files

### .yml/.yaml (Pipeline Configuration)

**What It Is:**
YAML files that define CI/CD pipelines (Azure Pipelines, GitHub Actions, etc.).

**Types in SupportServices:**

**1. OneBranch Pipelines**
```yaml
# .pipelines/build/OneBranch.Official.Chat.yml
trigger: none  # Manual trigger only

pool:
  image: 'onebranch.azurecr.io/windows/ltsc2022/vse2022:latest'
  os: windows

variables:
  BuildConfiguration: 'Release'
  BUILDDIR: '$(Build.SourcesDirectory)'

stages:
  - stage: Build
    displayName: 'Build Chat Domain'
    jobs:
      - job: BuildJob
        displayName: 'Build & Test'
        steps:
          - task: DotNetCoreCLI@2
            inputs:
              command: 'build'
              arguments: 'Chat\Chat.slnx -c Release /p:TreatWarningsAsErrors=true'
          
          - task: DotNetCoreCLI@2
            inputs:
              command: 'test'
              arguments: 'Chat\Chat.slnx'

  - stage: Deploy
    displayName: 'Deploy to Staging'
    dependsOn: Build
    jobs:
      - deployment: DeployStaging
        environment: 'Staging'
        strategy:
          runOnce:
            deploy:
              steps:
                - task: AzureResourceGroupDeployment@2
                  inputs:
                    resourceGroupName: '$(ResourceGroupName)'
                    location: 'eastus'
                    templateLocation: 'URL of template'
```

**2. Local Development Pipelines**
```yaml
# .pipelines/build/Build.NonOfficial.Chat.yml
trigger:
  - main
  - develop
  paths:
    - Chat/**

pr:
  - main
  - develop

pool:
  image: 'windows-latest'

stages:
  - stage: CompileScan
    displayName: 'Compile & Security Scan'
    jobs:
      - job: Compile
        steps:
          - task: DotNetCoreCLI@2
            inputs:
              command: 'build'
              arguments: 'Chat\Chat.slnx'
```

**Who Creates It:**
- **Platform team:** Initial setup
- **Service owners:** Modify for domain-specific builds
- **Developers:** Rarely edit; request changes via PR

## Package Management Files

### global.json (.NET SDK Version)

**What It Is:**
Specifies the EXACT .NET SDK version developers and CI/CD must use to build this project.

**Why It Exists:**
- ✅ **Consistency:** Everyone on same .NET version
- ✅ **Reproducible builds:** Same SDK = same output
- ✅ **Feature gating:** New language features only in specific versions

**SupportServices Example:**
```json
{
  "sdk": {
    "version": "10.0.204",
    "rollForward": "latestFeature"
  },
  "msbuild-sdks": {
    "Microsoft.Build.CentralPackageVersions": "2.1.3"
  }
}
```

**What Happens:**
```bash
# If developer has .NET 10.0.100 installed:
$ dotnet --version
10.0.100

# Run `dotnet build` in a repo with global.json specifying 10.0.204:
$ dotnet build
error: .NET SDK version 10.0.204 not found

# Solution:
dotnet sdk install
# or update build agent image
```

**Who Creates/Modifies:**
- **Platform team:** During setup
- **Developers:** Never touch unless upgrading .NET version
- **When:** Major .NET upgrades (annually)

### README.md (Project Overview)

**What It Is:**
Overview of what a project does, how to build/run/test it, key concepts.

**SupportServices Root Example:**
```markdown
# SupportServices Monorepo

A multi-domain .NET 10 backend for Xbox Support services.

## Domains

**What It Is:**

**SupportServices Example:**
```
# .editorconfig
root = true

# All files
[*]
charset = utf-8
insert_final_newline = true
trim_trailing_whitespace = true

# C# files
[*.cs]
indent_style = space
indent_size = 4
csharp_indent_case_contents = true
csharp_indent_switch_labels = true

# Naming conventions
[*.cs]
# Async methods: PascalCase ending with Async
dotnet_naming_rule.async_methods_should_be_pascal_case.symbols = async_methods
dotnet_naming_rule.async_methods_should_be_pascal_case.style = async_method_style
dotnet_naming_symbols.async_methods.applicable_kinds = method
dotnet_naming_symbols.async_methods.applicable_accessibilities = *
dotnet_naming_symbols.async_methods.required_modifiers = async
dotnet_naming_style.async_method_style.required_suffix = Async

# Private fields: _camelCase
dotnet_naming_rule.private_field_rule.symbols = private_fields
dotnet_naming_rule.private_field_rule.style = private_field_style
dotnet_naming_symbols.private_fields.applicable_kinds = field
dotnet_naming_symbols.private_fields.applicable_accessibilities = private
dotnet_naming_style.private_field_style.required_prefix = _

# JSON files
[*.json]
indent_style = space
indent_size = 2

## File Relationships & Workflows

### Dependency Graph

```
SupportServices.slnx (Root solution)
    ↓
Chat.slnx (Domain solution)
    ↓
Frontend.csproj (Project)
    ↓
Frontend.csproj inherits from:
  ├─ Directory.Build.props (global settings)
  │   ├─ Directory.Packages.props (versions)
  │   └─ Directory.Build.targets (post-build)
  ├─ nuget.config (package sources)
  └─ global.json (.NET SDK version)

At runtime:
  Frontend.csproj
    ├─ References appsettings.json (configuration)
    │   ├─ appsettings.Development.json (dev overrides)
    │   └─ appsettings.Production.json (prod overrides)
    ├─ References launchSettings.json (debug settings)
    └─ Uses .editorconfig (code style)

In CI/CD:
  .yml pipeline
    ├─ Reads global.json (ensures correct SDK)
    ├─ Runs: dotnet build (uses .csproj, .props, .targets)
    ├─ Deploys Bicep from Deploy/*.bicep
    └─ Uses azure.yaml (azd configuration)

In version control:
  .git/
    ├─ .gitignore (what to exclude)
    └─ .gitattributes (line ending rules)
```

### Build Process Flow

```
1. Developer runs: dotnet build Chat\Chat.slnx
   ↓
2. MSBuild reads:
   ├─ Chat.slnx (which .csproj files to build)
   ├─ Each Frontend.csproj reads:
   │  ├─ global.json (.NET SDK version)
   │  ├─ Directory.Build.props (common settings)
   │  ├─ Directory.Packages.props (package versions)
   │  └─ nuget.config (where to find packages)
   ↓
3. Compilation:
   ├─ Restore NuGet packages from Directory.Packages.props + nuget.config
   ├─ Compile C# files (enforce rules from .editorconfig)
   ├─ Run code analyzers (StyleCop, etc.)
   ├─ Generate IL code
   ↓
4. Post-build:
   ├─ Directory.Build.targets runs
   ├─ Copy files, generate version info, etc.
   ↓
5. Output:
   ├─ bin/Release/net10.0/Frontend.dll
   └─ bin/Release/net10.0/Frontend.deps.json
```

## When to Create, Modify, or Ignore

### Decision Matrix

| File | Create? | Modify? | When? | Who? |
|------|---------|---------|-------|-----|
| `.slnx` | ❌ Auto | ⚠️ Rarely | Adding projects | IDE or build system |
| `.csproj` | ❌ Auto | ⚠️ Rarely | Project-specific settings | Project owner |
| `Directory.Build.props` | ❌ Exists | ✅ Team decision | Updating global build rules | Platform team |
| `Directory.Packages.props` | ❌ Exists | ✅ Team decision | Security updates, major upgrades | Dependency manager |
| `global.json` | ❌ Exists | ✅ Team decision | .NET version upgrade | Quarterly review |
| `appsettings.json` | ✅ Per domain | ✅ Often | Adding config settings | Domain team |
| `appsettings.Production.json` | ❌ Never commit | ✅ Via Key Vault | CI/CD replaces | Deployment pipeline |
| `.gitignore` | ✅ Exists | ✅ Add patterns | New file types to exclude | Any developer |
| `.editorconfig` | ❌ Exists | ✅ Team decision | Updating code standards | Architecture team |
| `.yml` pipelines | ❌ Exists | ✅ Team decision | Pipeline changes | Build engineers |
| `README.md` | ✅ Per project | ✅ Keep updated | Documenting your work | Project owner |
| `CONTRIBUTING.md` | ✅ Root level | ✅ Team decision | Process changes | Leadership |

### File Creation Checklist

**When Adding a New Project:**
- [ ] Run `dotnet new classlib -n MyProject`
- [ ] Move to correct folder: `Domain/ProjectName/ProjectName.csproj`
- [ ] `.csproj` created ✅ (don't edit unless special cases)
- [ ] Add to solution: `Domain/Domain.slnx`

**When Configuring Application Settings:**
- [ ] Create `appsettings.json` in project root
- [ ] Add base settings (non-secret)
- [ ] Create `appsettings.Development.json` for local overrides
- [ ] Create `appsettings.Production.json` **locally only** (never commit)
- [ ] Or: Use Key Vault secrets in CI/CD (preferred)

**When Adding a NuGet Package:**
- [ ] Edit `Directory.Packages.props` (add version)
- [ ] In `.csproj`: `<PackageReference Include="Package.Name" />`
- [ ] Build and verify

**When Adding Build Rules:**
- [ ] Add to `Directory.Build.props` (not individual .csproj)
- [ ] Test across multiple projects
- [ ] Document in `CONTRIBUTING.md`

### Common Mistakes to Avoid

| Mistake | Impact | Prevention |
|---------|--------|-----------|
| Edit `.csproj` to set AssemblyName | Name inconsistency | Use Directory.Build.props auto-derivation |
| Commit `appsettings.Production.json` | Secrets leaked | Add to .gitignore; use Key Vault |
| Set package version in .csproj | Version mismatch across projects | Always use Directory.Packages.props |
| Ignore .editorconfig | Code style inconsistency | Use `dotnet format` before committing |
| Manually edit .sln/.slnx | Corruption; merge conflicts | Let IDE maintain; manual edits risky |
| Not update .gitignore | Artifact/secret leaks | Add patterns as you discover them |
| Hard-code connection strings | Security vulnerability | Use Configuration + Key Vault |

## Tips & Gotchas

### Pro Tips

**1. Always check what's inherited:**
```bash
# See what Directory.Build.props affects
dotnet msbuild /getProperty:AssemblyName

# See all imported properties
dotnet build --list-sdks
```

**2. Format before committing:**
```bash
dotnet format
git add .
git commit -m "chore: format code"
```

**3. Keep .gitignore updated:**
```bash
# See untracked files that should be ignored
git status --ignored

# Add new pattern
echo "build_artifacts/" >> .gitignore
```

**4. Validate your setup before complex work:**
```bash
# Verify SDK version
dotnet --version

# Verify packages restore correctly
dotnet restore --no-cache

# Verify build succeeds
dotnet build
```

---

### Gotchas

**Gotcha 1: AssemblyName/RootNamespace Auto-Derivation**
```csharp
// If you set these manually in .csproj, you break namespace consistency
// ❌ DON'T
<PropertyGroup>
  <AssemblyName>Wrong.Name</AssemblyName>
  <RootNamespace>Wrong.Namespace</RootNamespace>
</PropertyGroup>

// ✅ DO — let Directory.Build.props derive from folder path
// Folder: Refunds/Frontend/Frontend/
// Auto-derives: Microsoft.Xbox.Support.Refunds.Frontend
```

**Gotcha 2: Package Version Mismatches**
```xml
<!-- ❌ If you set versions in individual .csproj -->
<ItemGroup>
  <PackageReference Include="Newtonsoft.Json" Version="12.0.0" />
</ItemGroup>

<!-- While Directory.Packages.props says: -->
<PackageVersion Include="Newtonsoft.Json" Version="13.0.0" />

<!-- Result: CONFUSION. One or the other wins unpredictably -->

<!-- ✅ Always remove version from .csproj; rely on .props -->
<ItemGroup>
  <PackageReference Include="Newtonsoft.Json" />  <!-- Version from .props -->
</ItemGroup>
```

**Gotcha 3: Secrets in Configuration**
```json
// ❌ appsettings.json (committed to repo)
{
  "ConnectionStrings": {
    "CosmosDb": "AccountEndpoint=https://...;AccountKey=SECRET_KEY"
  }
}

// ✅ Use Key Vault or environment variables
{
  "KeyVaultUri": "https://myvault.vault.azure.net/",
  "ConnectionStrings": {
    "CosmosDb": "@secure-config:cosmos-key"  // Resolved at runtime
  }
}
```

**Gotcha 4: .sln/.slnx File Conflicts**
```bash
# If you accidentally have both .sln and .slnx in same folder
# Git will get confused. Remove the old one:
git rm --cached Old.sln
echo "Old.sln" >> .gitignore
git commit -m "Remove legacy .sln"
```

## Summary

### Key Takeaways

1. **Most files are auto-generated** — Let tooling create them; don't manually edit unless you know why
2. **Central management reduces complexity** — Directory.Build.props, Directory.Packages.props, .editorconfig
3. **Never commit secrets** — Use Key Vault, environment variables, or excluded files
4. **Use conventions** — .NET ecosystem has strong conventions; follow them
5. **Document your reasoning** — README.md, CONTRIBUTING.md explain to future you

- [ ] Ran `dotnet format` to enforce code style
